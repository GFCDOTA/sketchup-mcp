# FP-026 — Residual Wall Stub Elimination

> **This is a visual-fidelity bug:** the wall shell can be
> algorithmically valid and still architecturally wrong if endpoint
> caps / stubs remain visible.

## Status

Active (2026-05-27). First implementation pass landed alongside this
spec; remaining ambiguous candidates documented in the per-fixture
report.

## Context

The `planta_74` SKP generated by the `consensus.json → .skp` pipeline
showed repeated small wall leftovers at wall endpoints, corners,
junctions, and near openings. Visually they appear as small
rectangular protrusions attached to otherwise valid wall shells —
"toquinhos de sobra" in the user's words.

This is not a room-fidelity problem. It is a wall-shell
canonicalisation problem.

Related specs / lessons:

- **FP-024** — Windows as post-extrusion 3D apertures
- **FP-025 / LL-017** — Canonical wall shell: axis-aligned input must
  produce axis-aligned clean output (no L-shape notches, no slivers,
  no overhangs)
- **PR #192** — `_classify_endpoint_junctions` introduced
  junction-aware extension; eliminated the unconditional half-thickness
  extension at FREE endpoints. **First wave.** FP-026 closes the
  remaining residuals.

## Problem

The generated wall shell may contain residual geometry that is not
supported by the PDF or by the consensus model:

- dangling caps at external wall endpoints;
- small overhangs at L/T junctions;
- short rectangular protrusions near door/window cuts;
- wall pieces extending beyond intended intersections;
- repeated small blocks caused by segment-wise buffering or untrimmed
  square caps;
- carve-residue at opening edges when wall thickness varies.

These degrade architectural fidelity even when algorithmic gates pass
(`plan_shell_group_exists=True`, `wall_shell_is_single_group=True`,
`default_material_faces_zero=True`). Those metrics prove the shell
exists — not that it is visually clean.

## Definitions

**Wall stub** — a short protrusion, cap, or leftover wall piece
attached to a valid wall shell, where: its length is small relative
to the connected wall, it does not correspond to a wall segment in
the consensus, it is not visually supported by the PDF, and it
creates a visible "toquinho" in the SKP render.

**Residual wall cap** — a rectangular endpoint artifact created by
buffering/offsetting a wall centerline and failing to trim it against
an intersecting wall or opening.

**Valid wall return** — a short wall segment that is intentionally
present in the source geometry, PDF, or human annotation. Valid
returns must NOT be removed.

**Ambiguous protrusion** — render suggests a stub, but PDF/consensus
evidence is insufficient to safely delete. Reported as `WARN`, not
silently removed.

## Required debug stages

The builder must be debuggable across:

1. consensus wall centerlines (input);
2. raw wall rectangles (un-extended);
3. junction-aware wall rectangles (post-classifier, pre-union);
4. unioned wall shell (post `unary_union`);
5. wall shell after buffer-close-gap (`SNAP_EPS_PTS` round-trip);
6. wall shell after openings carve;
7. wall shell after canonicalise (collinear vertex drop);
8. final SKP extract (Ruby).

The diagnostic report identifies where each stub first appears.

## Detection heuristics

A candidate protrusion may be flagged when ALL of:

- protrusion length `< 1.5 × wall_thickness`;
- protrusion width approximately equal to wall thickness (±20%);
- attached to only one side of the shell (3 free edges, 1 attached);
- unsupported by consensus wall segment (no centerline passes
  through);
- not overlapping a valid door/window jamb region (opening
  `wall_id` ≠ candidate's touching wall);
- shell delta when the candidate is removed has area
  < threshold × total shell area.

Thresholds:
- `STUB_MAX_LENGTH_RATIO = 1.5` (× wall thickness)
- `STUB_WIDTH_TOLERANCE = 0.2` (relative to wall thickness)
- `STUB_MIN_ATTACHED_RATIO = 0.75` (fraction of free perimeter — flag if higher)

## PASS / WARN / FAIL

**PASS** — no visible residual stubs in top render; stub report
`fail = 0`; any remaining WARNs explicitly tied to ambiguous PDF
evidence; SKP + review renders committed under
`artifacts/review/<plant>/`.

**WARN** — a suspected stub exists but may correspond to a real PDF
detail; detection inconclusive; human/PDF evidence needed before
deletion; visually minor and not systematic.

**FAIL** — repeated residual caps appear across multiple walls;
wall extends beyond its intended junction with no PDF/consensus
support; opening carve leaves visible wall fragments; cleanup deletes
valid walls/doors/windows or breaks FP-024; claimed success leaves
`.skp` only under ignored `/runs/`.

## Correction strategy

Conservative, deterministic canonicalisation — NOT visual hacks:

1. **PR #192 baseline** (already landed): junction-aware endpoint
   extension via `_classify_endpoint_junctions` — extend only at
   endpoints meeting a perpendicular wall.
2. **Snap orthogonal intersections** — for endpoints within tolerance
   of a perpendicular wall's body, project to the wall edge before
   union.
3. **Merge collinear wall runs** — multiple collinear segments along
   the same centerline should produce a single rectangle, not stacked
   short pieces.
4. **Trim terminal overhangs after union** — detect protrusion
   primitives matching the stub heuristic; remove if classification
   = `residual_cap` or `overhanging_cap`.
5. **Cleanup post-carve residues** — after opening carves, drop short
   wall islands whose only support was the now-removed carve edge.
6. **Preserve valid returns** — anything with consensus centerline
   support stays; anything ambiguous flags WARN.

If a candidate is ambiguous, classify WARN and leave evidence — do
not silently delete.

## Implementation notes — likely root causes

After PR #192, remaining stubs may come from:

1. **`SNAP_EPS_PTS = 0.1` buffer-close-gap** — convex corners may
   leave 0.1 pt mitre artifacts.
2. **Different wall thicknesses meeting** — w000 (t=5.52) joining
   w001 (t=5.40) creates a 0.06 pt step.
3. **Opening carve mismatched widths** — carve rect may not perfectly
   align with wall thickness, leaving jamb slivers.
4. **Ruby exporter face splits** — `intersect_with` during 3D window
   aperture carve may leave extra edges.

Confirm whether the defect appears before or after Ruby/SketchUp
export via the staged diagnostic.
