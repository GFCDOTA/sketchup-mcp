# Quadrado Demo — Canonical Micro-Fixture Spec

> **Status:** Active reference (2026-05-23).
> **Purpose:** Smallest reproducible SU pipeline fixture, used as
> proof-of-concept laboratory for any operation that will later be
> applied to `planta_74_plan_shell`.
> **Rule lineage:** see Canonical Artifact Rule (user MEMORY
> `feedback_canonical_artifact_rule.md`, ROOT_RULE 2026-05-23). The
> quadrado is the **micro-fixture** of the 5-etapa flow:
> `micro-fixture → prova → regressão → planta → comparação`.

---

## 1. Why this fixture exists

Real plantas (e.g. `planta_74`) are too large + carry too many
incidental complications (multiple rooms, openings, soft barriers,
unit-conversion subtleties) for safe early experimentation. The
quadrado is a **single empty room** — the simplest topology that
still exercises:

- wall **shell/ring/footprint** (continuous, not 4 blocks touching)
- separate **floor layer** (z=0 face independent from walls)
- separate **ceiling layer** (z=height face independent)
- the **SU file format** (.skp) end-to-end
- the **autorun plugin mechanism** (`autorun_consume.rb` +
  `autorun_inspector_control.txt` lifecycle)

If an operation cannot succeed on the quadrado in isolation, it
will not succeed on `planta_74`. Conversely, success on the
quadrado is **necessary but not sufficient** — etapa 4 (regression
test) + etapa 5 (apply on planta + comparison) close the loop.

## 3. Geometric ground truth

### 3.1 Real-world dimensions (architect intent)

| Dimension | Value | Notes |
|---|---|---|
| Inner room | 4.00 m × 4.00 m × 2.70 m | from `consensus.json` walls in `inner_clear` mode |
| Wall thickness | 0.19 m | `wall_thickness_pts: 5.4` × `PT_TO_M = 0.19 / 5.4` |
| Outer envelope (intent) | 4.38 m × 4.38 m × 2.70 m | inner + 2 × thickness |
| Wall height | 2.70 m | hardcoded in `consume_consensus.rb` `WALL_HEIGHT_M` |
| Floor | one face at z = 0, footprint = outer ring | bottom of the wall shell |
| Ceiling | one face at z = wall height, footprint = outer ring | top of the wall shell |

### 3.2 Stored values in `quadrado.skp` (inspection ground truth)

The SKP was produced by `consume_consensus.rb` from `consensus.json`.
**The builder treats `walls.start/end` as centerlines regardless of
the declared `dimension_mode`** — this is a known builder
characteristic; see §7 caveat.

Inspection of `quadrado.skp` (via `_inspect_skp.rb` autorun):

```
path:                runs/quadrado_demo/quadrado.skp
file size:           32 KB
entity types:        Edge: 24, Face: 10, Group: 0
materials:           empty (Layer0 default)
layers:              1 (Layer0)
model bounds (in):   (134.78, 134.78, 0.0) → (299.74, 299.74, 106.30)
```

In meters (1 in = 0.0254 m):

| Face | y in inches | y in meters | Role |
|---|---|---|---|
| Outer south | 134.78 | 3.422 | -y boundary, points outward |
| Inner south | 142.26 | 3.613 | -y inner face |
| Inner north | 292.26 | 7.423 | +y inner face |
| Outer north | 299.74 | 7.613 | +y boundary, points outward |

Effective wall thickness (stored) = 142.26 - 134.78 = **7.48 in =
0.190 m** ✅.
Effective inner span = 292.26 - 142.26 = 150.00 in = **3.81 m**.

The 0.19 m discrepancy vs the consensus-declared 4.00 m inner is
the centerline-interpretation drift (see §7). Future work that
respects `dimension_mode: inner_clear` literally should aim for
the 4.00 m inner.

### 3.3 Topology (the shape that opens clean in SU)

The wall shell is **raw geometry on Layer0** — no `Sketchup::Group`,
no `Sketchup::ComponentInstance`. 34 entities total:

- 24 `Sketchup::Edge` instances forming the 4 outer + 4 inner
  vertical seams + 2 horizontal seams (top/bottom)
- 10 `Sketchup::Face` instances:
  - 4 outer side walls (at y=134.78, x=299.74, y=299.74, x=134.78)
  - 4 inner side walls (at y=142.26, x=292.26, y=292.26, x=142.26)
  - 1 bottom face at z=0 (annular: outer ring minus inner hole)
  - 1 top face at z=106.30 (annular: outer ring minus inner hole)

**Why this topology matters:** an SKP that opens cleanly in SU 2026
keeps faces raw on Layer0. The alternative —
`consume_consensus.rb`'s default with grouped walls + boolean
carving — produces a different structure that, when modified
incorrectly, exhibits the "abre o SU e fecha rápido" symptom (see
§7). Etapa-3 invariants (§5) explicitly check this topology stays
raw and continuous.

## 4. Coordinate model

### 4.1 Consensus (PDF-points domain)

`consensus.json` walls in `inner_clear` mode:

```json
{
  "wall_thickness_pts": 5.4,
  "dimension_mode": "inner_clear",
  "walls": [
    {"id": "w_bottom", "start": [100.0, 100.0],     "end": [213.684, 100.0],     "orientation": "h"},
    {"id": "w_top",    "start": [100.0, 213.684],   "end": [213.684, 213.684],   "orientation": "h"},
    {"id": "w_left",   "start": [100.0, 100.0],     "end": [100.0, 213.684],     "orientation": "v"},
    {"id": "w_right",  "start": [213.684, 100.0],   "end": [213.684, 213.684],   "orientation": "v"}
  ],
  "rooms": [{
    "id": "r_main", "name": "QUADRADO",
    "polygon_pts": [[102.7, 102.7], [210.984, 102.7],
                    [210.984, 210.984], [102.7, 210.984]]
  }]
}
```

Conversion: `PT_TO_M = 0.19 / 5.4 = 0.0352 m/pt`.

### 4.2 SKP (inch domain — SU internal unit)

SU stores everything in inches. `1 m = 39.3701 in`,
`PT_TO_IN = (0.19 / 5.4) × 39.3701 ≈ 1.385 in/pt`.

Wall midline at pt=100 → 100 × 1.385 = 138.5 in. **Stored outer
face is at 134.78 in** — that's the midline (138.52 in) minus
half-thickness (3.74 in) = 134.78 in ✅. Consistent with
centerline interpretation.

## 5. Invariants (8 hard rules)

A modification to the quadrado is **valid** iff all 8 invariants
hold relative to the canonical baseline. Validated by
`_invariants.py` (POC harness; see §6.5).

| # | Invariant | Description |
|---|---|---|
| I1 | `base_face_count == 10` | Baseline `quadrado.skp` has exactly 10 faces. |
| I2 | `modified_face_count > base_face_count` | A modification that ADDS geometry must increase the face count. |
| I3 | Floor preserved | The z=0 face (wall-ring footprint, annular) is unchanged in area. |
| I4 | Non-affected walls preserved | Walls that the diff does NOT touch must have identical face area. Default unaffected: north (y=299.74), east (x=299.74), west (x=134.78). |
| I5 | Target wall cut by expected area | The wall the diff targets loses an area equal to the diff's opening area (±50 in² tolerance). |
| I6 | Glass / fill face present at expected location | When the diff is a window, the glass face must sit at the mid-thickness y_mid (between outer and inner) with width matching the opening. |
| I7 | Model bounds unchanged | No origin shift, no scale change. `model.bounds.min` and `.max` must match the baseline exactly. |
| I8 | True through-hole on cuts | Both the outer AND the inner face of the target wall must lose the opening area — a "cut" that leaves the inner face intact is a fake cut (cosmetic, not a real opening). |

Each invariant has a corresponding assertion in `_invariants.py`.
A modification that fails any one of these is **rejected**.

### 6.4 Run a POC modification (in-place edit, window example)

```bash
"E:/Python312/python.exe" runs/quadrado_demo/_run_add_window.py
# Produces:
#   runs/quadrado_demo/quadrado_with_window.skp
#   runs/quadrado_demo/_add_window_log.txt
#   runs/quadrado_demo/_add_window_done.txt
```

Method (per `_add_window.rb`): SU opens `quadrado.skp` as
positional; autorun loads `_add_window.rb`; script:
1. **Reads y coordinates from the actual model** (LL-014, never
   hardcode).
2. Adds a coplanar rect face on the outer south face.
3. `entities.intersect_with` to force-split the outer face.
4. `pushpull` the resulting sub-face by the real wall thickness
   (NOT a theoretical constant) → SU auto-merges with the inner
   face if exact, creating a real through-hole.
5. If a back face remains, repeat steps 2–3 on the inner face +
   `erase_entities` of the resulting sub-face.
6. Add a glass face at y_mid with `alpha=0.45`, color
   `(140, 200, 230)`.
7. `model.commit_operation` + `model.save(out_path)`.

### 6.7 Safe SU lifecycle

All `_run_*.py` helpers under `runs/quadrado_demo/` launch SU as
a subprocess. They currently terminate SU when their done marker
appears. **This kills any concurrent human SU session.** The
canonical fix is the SU runner mode protocol (planned `tools/su_runner_safety.py`):
helpers respect `RUN_MODE` env (`interactive` = no terminate;
`headless` = terminate only own `proc.pid`). Until that lands on
develop, run these helpers only when no human SU session is open.

## 7. Known characteristics and caveats

### 7.1 Centerline interpretation despite `dimension_mode: inner_clear`

`consensus.json` declares `dimension_mode: inner_clear`, but the
`consume_consensus.rb` builder treats `walls.start/end` as
**centerlines** regardless. The resulting SKP has:

- Inner room: 3.81 m (instead of the declared 4.00 m)
- Outer envelope: 4.19 m (instead of 4.38 m)

This is **not a bug to fix in this spec** — it is a documented
characteristic of the existing builder. Future work that respects
`dimension_mode` literally should produce a 4.00 m inner room AND
update this spec accordingly. Both interpretations are useful as
laboratory cases.

### 7.2 The "rebuild" anti-pattern

`consume_consensus.rb` calls `model.entities.clear!` at the top of
`main()`. **This means you cannot use it to ADD a feature to an
existing SKP** — it discards the baseline and rebuilds from the
consensus. The validated in-place edit pattern (push/pull +
intersect_with on raw faces; §6.4) is the correct tool for adding
windows/doors/furniture to an existing baseline. This rule is the
core of the Canonical Artifact Rule (user MEMORY ROOT_RULE,
2026-05-23).

### 7.3 Float drift on `intersect_with`

`SU::Entities#intersect_with` requires float-exact coplanarity to
merge faces. Hardcoding wall coordinates from theoretical
conversion constants (e.g. `y_in = 134.784 + 7.5"`) introduces
drift that leaves vestigial slabs of 0.5 mm thickness in the
output, with 0.92 in² fragmentary faces that pass naive area
assertions but fail visual inspection. **Always read coordinates
from the model** (`face.bounds.min.y`); see §6.4 step 1.

### 7.4 SU subprocess lifecycle

Python launchers calling `Popen([SU, target_skp])` followed by
`proc.terminate()` will kill ANY running SU process for the user,
not just their own child (depending on OS signal propagation).
Future tooling must respect a `RUN_MODE` env (safe default:
`interactive` = do NOT terminate).

`runs/quadrado_demo/_invariants.py`:

| Invariant | Function in `_invariants.py` | Threshold |
|---|---|---|
| I1 | `int(base["face_count"]) == 10` | exact |

| I3 | `bottom_ok` (z=0 face area diff) | < 0.1 in² |
| I4 | `north_ok` (y=299.745 face area diff) | < 0.1 in² |
| I5 | `south_ok` (south outer area delta vs window area) | < 50 in² |
| I6 | `glass_ok` (glass face at y_mid with expected area) | < 50 in² |
| I7 | `base["model_bounds"] == mod["model_bounds"]` | byte equal |
| I8 | `inner_ok` (south inner area delta vs window area) | < 50 in² |

When the in-place edit pattern is promoted to a tracked tool
under `tools/`, these assertions should be promoted to
`tests/test_quadrado_window_invariants.py` (skip-if-missing for
CI compatibility, since `runs/` is gitignored).

## 9. Promotion path to `planta_74`

The 5-etapa flow:

| Etapa | Status | Artifact |
|---|---|---|
| 1. Micro-fixture canônico | ✅ | `runs/quadrado_demo/quadrado.skp` |
| 2. Prova isolada | ✅ | `runs/quadrado_demo/quadrado_with_window.skp` + render |
| 3. Teste/regressão/harness | 🟡 partial | `_invariants.py` (8/8 PASS); promote to `tests/` next |
| 4. Aplicação na planta real | ❌ | TODO: apply in-place edit to `runs/planta_74_plan_shell/model.skp` |
| 5. Comparação com baseline da planta | ❌ | TODO: SKP + render diff vs current planta_74 |

**Risk for etapa 4:** the planta_74 plan_shell SKP uses **groups**
with carved openings (not raw Layer0 faces like the quadrado). The
in-place edit method works on the active group's entities, not on
`model.active_entities`. The Ruby pattern must:

1. Identify the host wall **group**.
2. Enter the group's edit context (`group.entities`).
3. Apply the same `intersect_with` + `pushpull` pattern.
4. Validate the **group's** geometry (not the model root).

Etapa 4 must verify this transition before producing a comparable
SKP on planta_74. Until that verification ships, the quadrado POC
should NOT be claimed as "applicable to planta" — only "validated
in isolation on raw-Layer0 topology".
