# FP-014 P0 — spike documentado: `shapely.polygonize(walls)` não é viável no estado atual do consensus

> **Branch do spike:** `spike/fp014-p0-rooms-polygonize` (não mergeada)
> **Run base:** `runs/_milestone_skp_planta74_2026_05_09/` (post-merge `eef94bc`)
> **Hipótese testada:** "trocar `rooms_from_seeds` raster trace por
> `shapely.polygonize(walls)` + matchar seed→cell" (P0 do FP-014).
> **Resultado:** ❌ **não viável sem antes consertar
> `build_vector_consensus.py`** — walls + soft_barriers do consensus
> atual não formam um linework polygonizable.

## Sumário executivo

A hipótese P0 falhou. Polygonize retorna **0 a 9 cells**, nenhuma
contém qualquer seed de room. Não importa o snap tolerance, T-junction
split, virtual-walls em opening gaps, ou inclusão de soft_barriers
como linework — o consensus atual **não tem walls suficientes** para
fechar room cells.

**Não estou implementando o fix algorítmico** porque a hipótese P0
foi falsificada. Felipe explicitamente autorizou: *"Se o fix P0 for
grande demais, fazer primeiro um spike documentado com métricas e
critério de aceite."* Esse é o spike documentado.

## Métricas que falsificaram a hipótese

### Tentativa 1 — `polygonize(walls)` direto

```
walls=33, wall_thickness=5.40pt
unary_union(walls) → MultiLineString com 33 segments (preservados)
polygonize → 0 cells
```

Razão: walls têm **66 endpoints, todos únicos** (nenhum compartilhado
entre walls). Polygonize precisa de linework com endpoints
coincidentes para detectar ciclos.

### Tentativa 2 — snap + node + polygonize

```
snap_tol=0.5–10.0pt → max 3 cells, 0 seeds matched
union-find clustering → 66 endpoints → 39 unique nodes (snap 8.1pt)
polygonize → 0 cells
```

### Tentativa 3 — snap + T-junction split + polygonize

```
T-junction split: 33 walls → 36 walls (apenas 3 splits adicionados)
polygonize → 0 cells
```

### Tentativa 4 — virtual walls fechando opening gaps

```
7 colinear pairs com gap < 200pt detectados
+ 7 virtual walls inseridos
polygonize(33 + 7 = 40 walls) → 2 cells totalizando 0.2 m^2
0/11 seeds matched
```

### Tentativa 5 — `envelope - thick_walls`

```
walls buffered (thickness=5.4pt) área total: 14.5 m²
envelope bounding-box: 193 m²
envelope - walls: 1 single Polygon de 178.5 m²
```

Walls cobrem só **8% do envelope**. Interior - walls é UM ÚNICO
polígono, não cells separados. Walls **não dividem o envelope** em
rooms.

### Tentativa 6 — incluir soft_barriers como linework

```
walls + soft_barriers = 179 segments totalizando 280.7 m
unary_union → MultiLineString com 204 segments (após noding shapely)
polygonize → 9 cells totalizando 16.0 m^2
0/11 seeds matched (todas as 9 cells são fragmentos pequenos)
```

Soft_barriers contêm polylines complexas (sb000 tem 30 vértices
formando boa parte do envelope), mas mesmo combinados com walls,
**polygonize gera apenas 9 cells pequenas** (max 3.58 m²) — os
rooms reais (10–26 m²) permanecem como interior aberto.

## Diagnóstico técnico

**O `tools/build_vector_consensus.py` extrai walls como segments
isolados sem topologia de rede.** Características:

1. Cada wall = `LineString` com `start`, `end` independentes
2. Não compartilha endpoints com walls adjacentes (66 endpoints
   únicos entre 33 walls)
3. Não tem T-junctions explícitas (extractor não os detecta)
4. Walls externos da torre (envelope) são **classificados como
   soft_barriers** (peitoris com vidro), não walls
5. Walls divisórias internas que não satisfazem o filtro
   "min(width,height) ≈ wall_thickness" podem estar **ausentes
   completamente**

**Implicação:** o output do detector é adequado para o método raster
trace existente (que usa walls como obstáculos para watershed
flood-fill), mas **inadequado para qualquer abordagem
polygon-baseada** (`polygonize`, `cell decomposition`, `face
extraction`). A topologia simplesmente não está lá.

## Por que `rooms_from_seeds` raster trace produz polygons defeituosos

Mesmo com input incompleto, o raster trace funciona — gera polygons
de room. O problema é a **qualidade**:

- Watershed flood-fill enche a região onde não há walls
- Concave-hull bound (sobre wall endpoints) só marca o envelope
  externo, não as divisórias entre rooms
- Resultado: SUITE 01 polygon vaza atrás de BANHO 02 porque NENHUMA
  wall fecha a divisão entre os dois — o wall divisório existe no PDF
  mas não foi extraído pelo `build_vector_consensus.py`

Trocar raster trace por polygonize **não resolve** se o input é o
mesmo.

## Caminhos viáveis (decisão necessária)

### Opção α — refatorar `tools/build_vector_consensus.py` (BIG)
Investigar e ampliar o detector para:
- Detectar walls divisórias internas que o filtro atual perde
- Manter walls externos como walls (não soft_barriers) OU tratar
  soft_barriers como walls estruturais para fins de polygonize
- Garantir endpoints compartilhados via snap durante extração
- Adicionar T-junctions explícitas

**Esforço:** ~500 LOC de refactor + ~50 testes; risco médio
(detector é gate cedo, mudança propaga). Pode requerer ADR.

### Opção β — melhorar `tools/rooms_from_seeds.py` raster trace (MID)
Manter raster trace, mas:
- Pre-filter polygons gerados por simplicidade (rejeitar polygons
  com >50 vts; refinar via convex/concave hull mais agressivo)
- Adicionar passo "snap to nearest wall" pra cada vértice
- Truncar polygon onde vertex está > N pt de qualquer wall

**Esforço:** ~150 LOC + ~20 testes. Não resolve "polygon vaza
atrás de wall ausente" — apenas suaviza visualmente.

### Opção γ — só Opção C (gate F0 estrutural) + adiar refactor (CONSERVADOR)
Não tocar detector nem rooms_from_seeds agora. Implementar **só
o gate F0** que bloqueia SKP defeituoso. Aceitar que SKP da
planta_74 não exporta enquanto detector não melhorar.

**Esforço:** ~150 LOC + ~15 testes. Honest sobre o estado atual:
F0 vai sempre FAIL para planta_74 até detector ser refeito.

## Critérios de aceite (para qualquer opção que avançar)

Mantidos os critérios definidos por Felipe:
1. Rooms/floors não podem vazar para fora do envelope
2. Não pode haver triângulo gigante na área direita (SUITE 01)
3. Polígonos precisam ser células plausíveis delimitadas por walls
4. F0 ou teste equivalente precisa detectar regressão FP-014
5. SKP novo precisa ser visualmente melhor que o atual

**Critério adicional descoberto neste spike:**
6. Detector deve produzir walls com endpoints compartilhados
   (topologia explícita) OU rooms_from_seeds deve aceitar input
   sem topologia e produzir polygons honest sem vazamento
   (mais difícil)
