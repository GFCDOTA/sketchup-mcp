## Goal of this investigation

Cycle 11b was supposed to:
- author a second `ground_truth/<plant>/expected_model.json`
- using `planta_74_clean.pdf` as a same-family benchmark (per
  Felipe's framing: "not a generalization proof, but useful for
  testing variation, GT stability, fidelity engine robustness,
  and clean-vs-contaminated PDF behaviour")
- run the fidelity engine against it
- ship a comparison report `planta_74` vs `planta_74_clean`

`tools.build_vector_consensus` rejects `planta_74_clean.pdf`:

```
      drawings=0 page_size=856x1212 mode=raster-like.
```

Same outcome for `test_plan.pdf` (which is generated by
`make_test_pdf.py` using PIL, which writes raster-image PDFs).

## Why this matters

The Fidelity Engine v1, the Plan Truth Gate, the Micro Truth
Gate, and the smoke harness are all built around the vector
pipeline. The raster pipeline (`extract/`, `classify/`,
`topology/`, `model/`) carries 16 known failing tests
(documented in CLAUDE.md §10) and is OUTDATED relative to
vector — the vector path is the supported route.

Therefore: **a second `expected_model.json` requires a second
vector PDF**. Repo currently has none.

## Why I'm NOT building a workaround in this PR

Three options were considered:

1. **Convert `planta_74_clean.pdf` → vector via OCR/tracing.**
   Out of scope — non-trivial (would need new dep like
   `pdf2image` + `potrace` + reapproximation), and the vector
   topology produced by tracing wouldn't match what an
   architectural CAD export looks like. The fidelity engine
   would not see a clean signal.

2. **Use the raster pipeline.** Documented as outdated in
   CLAUDE.md §10 (16 tests fail in main due to gate
   `len(strokes) > 200` in `classify/service.py:160`).
   Investing here is throwing good time after a deprecated
   path.

3. **Build a synthetic vector PDF generator.** This is the
   right next step (Cycle 11c, queued). Requires a new dev
   dependency (e.g., `reportlab`) OR ~100 LOC of hand-written
   PDF content stream. Bigger scope than this Cycle 11b's
   docs-only deliverable; deserves its own PR with new tests.

## What this PR delivers

- This diagnostic doc.
- Inventory entry in `.ai_bridge/TODO_NEXT.md` queueing Cycle 11c
  with a clear scope.
- No code changes. No new test surface. No new dependency.

## Next step (Cycle 11c, NOT in this PR)

**`tools/synth/make_synthetic_vector_pdf.py`** — generates a
small (2-room or L-shape) vector PDF from a tiny
`expected_model.json` description. Authoring this tool means:

- a vendor pick: `reportlab` (mature, ~6 MB install) vs
  hand-rolled (no dep, ~100 LOC of PDF content stream)
- output schema: minimal vector with 4-8 walls, 2-3 rooms,
  1-2 openings, 2 text labels for room seeds
- end-to-end round-trip test: synth -> pipeline -> classify ->
  fidelity should return 1.0 (engine self-check on REAL
  pipeline output, not just on engine-internal helpers as
  `synth_from_expected.py` does today)

This is the proper foundation for multi-PDF benchmark work.
True generalization (5+ different real plantas from independent
sources) still requires Felipe to provide PDFs — that's a
separate RED-blocked item.
